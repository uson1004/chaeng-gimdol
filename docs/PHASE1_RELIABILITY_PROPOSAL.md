# 챙김돌 Phase 1 신뢰성 병목 검토 제안서

## 요약

현재 가장 큰 병목은 백엔드 부재가 아니라 **실사용 환경에서 위치 감지와 알림이 믿을 만큼 동작하는지 검증되지 않은 점**이다.

챙김돌 Phase 1은 Android-only, local-first MVP다. README와 프로젝트 작업 계약 모두 Phase 1에서 서버, FCM, LOST112, 날씨, 블루투스 태그, 백그라운드 위치를 제외한다. 따라서 지금 필요한 일은 서버 설계가 아니라, 로컬 앱이 실제 외출 상황에서 사용자에게 약속한 최소 가치를 안정적으로 제공하는지 확인하는 것이다.

## 프로젝트 성격 재확인

챙김돌의 핵심 가치는 “외출 직전 물건을 확인하고, 사용자가 직접 시작한 세션 동안 이동이 감지되면 한 번 더 확인하게 돕는 것”이다.

현재 MVP 범위:

- 기본 물품 온보딩
- 물품 추가, 활성화, 삭제
- 수동 체크리스트 세션
- 세션 중에만 실행되는 위치 포그라운드 서비스
- 10분 이후 시작점에서 100m 이상 이동 시 세션당 1회 알림
- 위치/알림 권한 없이도 가능한 수동 완료 흐름
- 로컬 데이터 전체 삭제

명시적 제외 범위:

- 서버/백엔드
- FCM
- LOST112
- 날씨
- 블루투스 태그
- 백그라운드 위치
- raw location 좌표 저장

결론: Phase 1의 성공 조건은 “서버가 있는가”가 아니라 “로컬 앱만으로 수동 체크와 포그라운드 이동 알림이 실제로 믿을 만한가”다.

## 병목 1. 실기기 위치·알림 신뢰성 검증

### 현재 상태

자동 테스트는 통과하지만, `docs/MANUAL_QA.md`에는 API 26/33/36 환경, 실제 이동, 권한 거부/복구, 알림 탭 복귀가 미실행 항목으로 남아 있다.

이 병목은 제품 핵심 가치에 직접 닿는다. 사용자가 앱을 켜고 외출했는데 알림이 오지 않거나, 알림을 눌러도 진행 중 세션으로 돌아오지 못하면 백엔드 유무와 무관하게 MVP는 실패한다.

### 제안

1. API 26, API 33, API 36, 최신 AVD 또는 실기기 1대에서 수동 QA를 실행한다.
2. 최소 한 번은 실제 실외 이동 테스트를 수행한다.
3. 권한 상태별 시나리오를 고정한다.
   - 위치 권한 거부
   - 대략적 위치만 허용
   - 알림 권한 거부
   - 권한 재허용
   - 설정에서 출발 감지 off
4. 알림 탭 복귀를 검증한다.
   - 진행 중 세션 알림
   - 이미 완료된 세션 알림

### 완료 기준

- 위치/알림 권한이 없어도 수동 체크가 막히지 않는다.
- 위치 감지 중 포그라운드 알림이 보인다.
- 알림을 누르면 진행 중 세션으로 이동한다.
- 완료된 세션 알림을 누르면 홈에서 종료 안내를 보여준다.
- 실행 중 세션은 앱 재시작 후에도 이어갈 수 있다.

## 병목 2. 포그라운드 위치 서비스의 현실 조건

### 현재 상태

`LocationTrackingService`는 로컬 메모리의 시작 위치를 기준으로 동작한다. 정책은 `ExitDetectionPolicy`에 분리되어 있으며 현재 조건은 다음과 같다.

- 최소 10분 경과
- 시작점에서 100m 초과 이동
- 위치 정확도 50m 이하
- 세션당 알림 1회

이 설계는 Phase 1의 개인정보 원칙과 맞다. raw location 좌표를 Room이나 로그에 저장하지 않고, 서비스 생존 중 메모리에서만 판단한다.

### 병목 판단

서버는 이 문제를 해결하지 못한다. Android 권한, 위치 정확도, 포그라운드 서비스 유지, 배터리 정책, 제조사별 제한은 모두 클라이언트 현실 조건이다.

### 제안

1. 정책값은 지금 유지한다.
   - 10분/100m/50m는 MVP 기준으로 충분히 단순하고 설명 가능하다.
2. 실제 이동 QA에서 실패 케이스만 기록한다.
   - 알림 미발생
   - 너무 이른 알림
   - 정확도 낮은 위치에서 오탐
   - 서비스가 조용히 죽는 경우
3. 실패가 반복될 때만 최소 진단 정보를 추가한다.
   - 세션 ID
   - 서비스 시작/중지 이벤트
   - 알림 발송 여부
   - 정책 판단 결과 코드
4. 좌표 저장은 계속 금지한다.

### 완료 기준

- 10분 전에는 알림이 오지 않는다.
- 100m 미만 이동에는 알림이 오지 않는다.
- 정확도 50m 초과 위치는 이탈로 판단하지 않는다.
- 한 세션에서 이동 알림은 최대 한 번만 온다.
- 세션 완료 또는 설정 off 시 서비스가 중지된다.

## 병목 3. 사용자 가치 검증

### 현재 상태

현재 앱은 기능적으로 MVP 흐름을 갖췄다. 남은 질문은 “실제로 외출 전에 쓰는가”다.

서버, 계정, 동기화는 이 질문에 답하지 않는다. 오히려 초기 사용 저항을 높일 수 있다.

### 제안

5~10명의 소규모 사용자에게 3일간 사용을 요청한다.

테스트 조건:

- 새 설치부터 시작
- 자주 챙기는 물건 3~6개 등록
- 실제 외출 전 최소 1회 챙김 세션 수행
- 위치 감지는 원하는 사용자만 켬
- 사용 후 3문항 회고 수집

회고 질문:

1. 외출 전에 실제로 앱을 열었는가?
2. 체크리스트가 빠뜨림 불안을 줄였는가?
3. 이동 알림이 도움이 되었는가, 귀찮았는가?

### 완료 기준

- 5명 중 3명 이상이 실제 외출 전 세션을 1회 이상 사용한다.
- 사용자가 직접 등록한 물건이 기본 템플릿보다 늘어난다.
- 이동 알림은 “유용함” 또는 “거슬리지 않음”으로 평가된다.
- 사용자가 서버/계정 부재를 핵심 불편으로 먼저 말하지 않는다.

## 백엔드 판단

### 지금은 보류

Phase 1에서는 백엔드를 만들지 않는다.

이유:

- 현재 명시 범위에서 제외되어 있다.
- 핵심 병목이 서버로 해결되지 않는다.
- raw location 좌표 저장 금지 원칙과 충돌 위험이 있다.
- 계정/동기화는 초기 MVP 사용 장벽을 높인다.

### 백엔드가 필요한 조건

아래 조건 중 하나가 실제 사용자 피드백으로 확인될 때 Phase 2에서 검토한다.

- 여러 기기 간 물품 목록 동기화 요구
- 기기 변경 시 백업/복원 요구
- 가족/동거인 공유 체크리스트 요구
- LOST112, 날씨, 공공데이터 연동 요구
- 원격 푸시 또는 리마인더 정책 서버 제어 필요
- 익명 장애 리포트 수집 필요

## 2주 실행안

### 1주차: 신뢰성 QA

- API 26/33/36 또는 대체 실기기 조합 준비
- `docs/MANUAL_QA.md` 기준 수동 QA 실행
- 실제 이동 테스트 2회 이상 수행
- 실패 항목만 이슈화

### 2주차: 사용자 가치 검증

- 5~10명에게 3일 사용 요청
- 외출 전 사용 여부와 알림 만족도 수집
- 서버/계정 요구가 자연 발생하는지 확인
- Phase 2 백엔드 필요 여부 재판단

## 최종 권고

지금의 최단 경로는 백엔드가 아니라 **실기기 QA와 소규모 사용자 검증**이다.

서버는 사용자가 “여러 기기/공유/백업/외부 데이터”를 명확히 요구할 때 추가한다. 그 전까지는 로컬 신뢰성을 먼저 닫는 것이 챙김돌 Phase 1에 맞는 가장 작은 해법이다.
