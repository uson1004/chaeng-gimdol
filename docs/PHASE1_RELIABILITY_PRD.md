# 챙김돌 Phase 1 신뢰성 검증 PRD

## 상태

- 작성일: 2026-07-02
- 기준 문서: `docs/PHASE1_RELIABILITY_PROPOSAL.md`
- 범위: Android-only, local-first Phase 1 MVP
- 결정: 백엔드는 Phase 1에서 보류하고 실기기 신뢰성 검증과 사용자 가치 검증을 먼저 완료한다.

## 문제 정의

챙김돌의 핵심 가치는 외출 직전 사용자가 물건을 확인하고, 사용자가 직접 시작한 챙김 세션 중 이동이 감지되면 한 번 더 확인하도록 돕는 것이다.

현재 자동 테스트와 에뮬레이터 검증은 통과했지만, API 26/33/36, 실제 이동, 권한 거부와 복구, 알림 탭 복귀 같은 실사용 조건은 아직 수동 검증이 끝나지 않았다. 서버를 추가해도 Android 권한, 위치 정확도, 배터리 정책, 포그라운드 서비스 유지 문제는 해결되지 않는다.

## 목표

1. 실기기 또는 대체 에뮬레이터 조합에서 위치와 알림 신뢰성을 검증한다.
2. `10분 / 100m / 정확도 50m / 세션당 1회` 정책이 실제 이동 조건에서 과도하거나 부족하지 않은지 확인한다.
3. 5~10명의 실제 사용자가 외출 루틴에서 챙김 세션을 여는지 검증한다.
4. 백엔드가 필요한 조건을 실제 피드백 기반으로만 Phase 2 후보에 올린다.

## 비목표

- 서버, 계정, 동기화, FCM
- LOST112, 날씨, 공공데이터 연동
- 블루투스 태그
- 백그라운드 위치 권한
- raw location 좌표 저장
- 디자인 개편 또는 신규 화면 대규모 추가

## 사용자 시나리오

### 외출 전 수동 확인

사용자는 홈에서 챙김 세션을 시작하고, 물품을 `미확인`, `챙김`, `해당 없음`으로 구분한 뒤 세션을 완료한다. 위치 또는 알림 권한이 없어도 이 흐름은 막히면 안 된다.

### 이동 중 재확인

사용자가 위치 감지를 켠 상태로 세션을 시작하면 앱은 포그라운드 서비스 알림을 표시한다. 세션 시작 10분 이후 시작점에서 100m 초과 이동했고 위치 정확도가 50m 이하일 때, 세션당 한 번만 재확인 알림을 보낸다.

### 알림 복귀

사용자가 진행 중 세션의 포그라운드 알림이나 재확인 알림을 누르면 관련 세션으로 돌아와야 한다. 이미 끝난 세션 알림을 눌렀다면 홈에서 종료 상태를 이해할 수 있어야 한다.

## 요구사항

### R1. 환경 매트릭스 검증

- API 26, API 33, API 36 환경을 준비해 `docs/MANUAL_QA.md`를 실행한다.
- 시스템 이미지 준비가 어려운 경우 동일 Android 버전 실기기 또는 최신 AVD 1대를 대체 증거로 기록한다.
- 각 실행 결과는 통과, 실패, 미실행으로 남긴다.

### R2. 권한 상태 검증

- 알림 권한 거부 상태에서도 수동 체크가 가능해야 한다.
- 위치 권한 거부 상태에서는 위치 감지를 시작하지 않고 수동 흐름을 유지해야 한다.
- 대략적 위치만 허용해도 앱이 종료되면 안 된다.
- 권한을 다시 허용하면 다음 세션부터 감지가 가능해야 한다.
- 설정에서 출발 감지를 끄면 실행 중 서비스가 중지되어야 한다.

### R3. 위치 정책 검증

- 세션 시작 10분 전에는 이동 알림이 오지 않아야 한다.
- 시작점에서 100m 이하 이동에는 이동 알림이 오지 않아야 한다.
- 위치 정확도 50m 초과 값은 이탈로 판단하지 않아야 한다.
- 한 세션에서 이동 알림은 최대 한 번만 발생해야 한다.
- 세션 완료 또는 감지 설정 off 시 서비스가 중지되어야 한다.

### R4. 개인정보 제약

- 위도와 경도는 Room, DataStore, 로그, 문서화된 수집 항목에 저장하지 않는다.
- 실제 이동 QA에서 실패를 기록할 때도 좌표 대신 세션 ID, 서비스 시작/중지, 알림 발송 여부, 정책 판단 결과처럼 좌표가 아닌 진단 정보만 사용한다.

### R5. 사용자 가치 검증

- 5~10명에게 3일간 실제 외출 전 사용을 요청한다.
- 각 사용자는 새 설치 상태에서 자주 챙기는 물건 3~6개를 등록한다.
- 위치 감지는 선택으로 둔다.
- 사용 후 아래 질문을 수집한다.
  - 외출 전에 실제로 앱을 열었는가?
  - 체크리스트가 빠뜨림 불안을 줄였는가?
  - 이동 알림이 도움이 되었는가, 귀찮았는가?

## 완료 기준

- `docs/MANUAL_QA.md`의 핵심 흐름, 권한, 위치와 알림, 접근성, 데이터와 개인정보 항목 결과가 기록되어 있다.
- API 26/33/36 또는 문서화된 대체 환경에서 수동 QA 결과가 남아 있다.
- 실제 이동 테스트가 2회 이상 수행되었고 실패 항목은 재현 조건과 함께 이슈화되어 있다.
- 5명 이상 사용자 검증 결과가 남아 있다.
- 5명 중 3명 이상이 실제 외출 전 챙김 세션을 1회 이상 사용했다.
- 서버/계정 부재가 핵심 불편으로 반복 보고되지 않았다.

## 백엔드 재검토 조건

아래 요구가 사용자 피드백으로 반복 확인될 때 Phase 2 백엔드를 검토한다.

- 여러 기기 간 물품 목록 동기화
- 기기 변경 시 백업과 복원
- 가족 또는 동거인 공유 체크리스트
- LOST112, 날씨, 공공데이터 연동
- 원격 푸시 또는 서버 제어 리마인더
- 좌표를 저장하지 않는 익명 장애 리포트

## 실행 계획

### 1주차: 신뢰성 QA

- API 26/33/36 또는 대체 실기기 조합 준비
- `docs/MANUAL_QA.md` 기준 수동 QA 실행
- 실제 이동 테스트 2회 이상 수행
- 실패 항목만 이슈화하고 좌표 없는 진단 정보로 정리

### 2주차: 사용자 가치 검증

- 5~10명에게 3일 사용 요청
- 외출 전 사용 여부와 알림 만족도 수집
- 서버/계정 요구가 자연 발생하는지 확인
- Phase 2 백엔드 필요 여부 재판단

## 출시 판단

Phase 1 출시 판단은 백엔드 구현 여부가 아니라, 로컬 수동 체크와 포그라운드 이동 알림이 실제 외출 조건에서 신뢰 가능한지에 따라 결정한다.
